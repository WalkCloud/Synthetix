# Synthetix - 智能文档编写工具 需求分析文档

**版本**: v1.0.0
**日期**: 2026-05-02
**状态**: 需求分析阶段

---

## 一、项目概述

### 1.1 项目背景

在知识密集型工作中，文档编写是一项高频且耗时的任务。现有的文档工具缺乏智能辅助能力，无法有效利用已有知识库进行内容生成，同时长文档编写面临上下文溢出和内容污染问题。Synthetix 旨在构建一款支持离线和云端部署的智能文档编写工具，通过 RAG 检索、LLM 生成、章节化编写等能力，帮助用户高效产出高质量文档。

### 1.2 项目目标

| 目标 | 描述 |
|------|------|
| 统一文档格式 | 将多源异构文档转换为标准化 Markdown 格式 |
| 知识检索 | 通过 RAG 技术实现文档内容的语义检索 |
| 智能辅助 | 借助 LLM 服务辅助思路整理和大纲生成 |
| 章节化编写 | 分章节逐步生成，解决长文档上下文溢出问题 |
| 双模型对比 | 支持双 LLM 服务同步生成，用户择优选用 |
| 灵活部署 | 支持本地离线运行和云端在线部署两种模式 |

### 1.3 目标用户

- 技术文档编写者（架构文档、API 文档、技术方案）
- 研究人员（论文、研究报告）
- 产品经理（PRD、竞品分析）
- 企业知识管理人员

### 1.4 运行模式

| 模式 | 存储 | LLM 服务 | 适用场景 |
|------|------|----------|----------|
| 离线本地服务模式 | 本地文件系统 / SQLite 或 PostgreSQL | Ollama 本地模型 / 用户自配 API | 个人本机、企业内网、数据敏感场景 |
| 离线 Docker 模式 | 本地文件系统 / MinIO / PostgreSQL | Ollama 本地模型 / 用户自配 API | 技术用户、小团队、服务器内网部署 |
| 云端模式 | S3 对象存储 | 公有云 API | 团队协作、多端访问 |

### 1.5 产品定位与建设原则

本项目的核心定位不是通用文档编辑器，而是一个“可追溯、可控、可分段的长文档生成工作台”。第一阶段应优先打通从资料上传、知识库初始化、思路澄清、大纲生成、章节化写作到引用来源追踪的核心闭环。

建设原则如下：

| 原则 | 说明 |
|------|------|
| 核心闭环优先 | 优先完成文档初始化、RAG 检索、思路整理、章节生成和引用追踪，不在早期过度扩展为完整 Office 平台 |
| 可信生成优先 | 生成内容必须能追溯参考来源，保留引用片段、来源文档、章节、页码或块级锚点 |
| 章节化状态管理 | 长文档生成必须按章节管理状态、摘要和引用关系，避免上下文溢出和上下文污染 |
| 服务解耦 | Next.js 负责产品界面和业务 API，文档转换、RAG 索引、LLM 编排等长任务应由独立 worker 或服务承载 |
| 能力可插拔 | LLM、Embedding、Reranker、Vision、文生图、图表生成等能力应通过模型能力标签和 Skill/MCP 编排接入 |
| 离线优先兼容云端 | 核心能力应能在离线本地服务模式运行，云端部署作为扩展能力支持团队协作和多端访问 |
| 不做桌面安装版 | 产品不采用 Tauri/Electron 等桌面壳路线，离线使用方式以本地 Web 服务为主 |

| F0-US4 | 作为用户，我想在仪表盘搜索文档并跳转到文档库查看结果 | P1 |

#### 功能描述

#### 用户故事

| 编号 | 用户故事 | 优先级 |

| NT-US1 | 作为用户，我想在文档转换完成或失败时收到通知 | P0 |

### 2.2 F1 - 文档初始化

#### 2.2.1 功能描述

用户上传各类格式的文档，系统将其转换为统一的 Markdown 格式，存储并同步向量化以支持 RAG 检索。

#### 2.2.2 用户故事

| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| F1-US1 | 作为用户，我想上传 PDF/Word/PPT/Excel/HTML 等格式的文档，系统自动转换为 Markdown | P0 |
| F1-US2 | 作为用户，我想选择将转换后的文档存储在本地目录或远端 S3 存储服务 | P0 |
| F1-US3 | 作为用户，我想让大文档自动拆分为合理的子文档，避免超出模型上下文限制 | P0 |
| F1-US4 | 作为用户，我想让文档内容自动向量化存储，以支持后续语义检索 | P0 |
| F1-US5 | 作为用户，我想配置使用 Ollama 本地嵌入模型或公有云嵌入模型 | P1 |

#### 2.2.3 功能流程

```
用户上传文档
    │
    ▼
格式检测与校验
    │
    ▼
MarkItDown 转换为 Markdown
    │
    ▼
文档大小检测 ──── 文档 > 模型上下文 50%? ──是──▶ LLM 智能拆分
    │                                                        │
    否                                                       │
    │◄───────────────────────────────────────────────────────┘
    ▼
存储至用户指定位置（本地/S3）
    │
    ▼
LightRAG 向量化存储
    │
    ▼
初始化完成，返回文档元数据
```

#### 2.2.4 业务规则

| 编号 | 规则描述 |
|------|----------|
| F1-BR1 | 支持的输入格式：PDF、DOCX、PPTX、XLSX、HTML、EPUB、图片（OCR） |
| F1-BR2 | 当 Markdown 文档大小超过用户选定 LLM 模型上下文窗口的安全阈值时，触发自动拆分。阈值默认 50%，用户可在 40%-60% 范围内自定义调整 |
| F1-BR3 | 拆分策略：按领域、主题、相关性进行语义拆分，保持内容完整性 |
| F1-BR4 | 拆分后的子文档各自独立存储并向量化 |
| F1-BR5 | 嵌入模型支持两种模式：Ollama 本地服务 / 公有云 API（OpenAI、Cohere、Voyage、Jina、云厂商兼容嵌入服务等） |
| F1-BR6 | 向量化过程中保留文档元数据（来源、作者、创建时间、拆分关系） |
| F1-BR7 | 单文件上传大小限制默认为 100MB，可通过配置调整 |
| F1-BR8 | 支持上传重复文件检测（基于文件哈希），重复时提示用户选择覆盖或跳过 |

#### 2.2.5 文档拆分与索引策略

大文档拆分不应只依据字符数，应以模型 token 上下文为主要判断依据，并预留系统提示词、用户约束、参考片段和输出内容的安全空间。

| 策略 | 说明 |
|------|------|
| Token 估算 | 根据用户选定模型的 context window 估算文档 token 数，默认在 40%-50% 安全阈值触发拆分 |
| 结构优先 | 优先按标题层级、目录、页、章节、表格边界进行确定性拆分 |
| LLM 修正 | 对结构拆分后的片段再按领域、主题、相关性进行合并或重切 |
| 元数据保留 | 拆分后的每个子文档必须保留原始文件、页码、标题路径、块 ID、图片来源等 provenance 信息 |
| 可人工调整 | 对拆分不合理的文档，应允许用户重新拆分、合并子文档或重新索引 |

#### 2.2.6 技术依赖

| 依赖 | 用途 | 版本要求 |
|------|------|----------|
| [MarkItDown](https://github.com/microsoft/markitdown) | 文档格式转换 | 最新稳定版 |
| [LightRAG](https://github.com/hkuds/lightrag) | 文档向量化与 RAG 检索 | 最新稳定版 |
| Ollama（可选） | 本地嵌入模型服务 | v0.1.x+ |

#### 2.2.7 接口设计

```
POST   /api/v1/documents/upload        上传文档
GET    /api/v1/documents                 查询文档列表
GET    /api/v1/documents/:id             查询文档详情
DELETE /api/v1/documents/:id             删除文档
GET    /api/v1/documents/:id/status      查询文档处理状态
POST   /api/v1/documents/:id/reprocess   重新处理文档
```

### 2.3 F2 - 文档库管理

#### 2.3.1 功能描述

用户可浏览、检索、管理初始化后的文档库，支持关键词搜索和语义检索两种方式。

#### 2.3.2 用户故事

| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| F2-US1 | 作为用户，我想浏览文档库中所有已转换的 Markdown 文档 | P0 |
| F2-US2 | 作为用户，我想通过关键词搜索文档标题和内容 | P0 |
| F2-US3 | 作为用户，我想通过自然语言进行语义检索，找到相关文档片段 | P0 |
| F2-US4 | 作为用户，我想查看文档的元信息（来源、格式、大小、拆分关系） | P1 |
| F2-US5 | 作为用户，我想对文档进行分类管理和标签标记 | P1 |
| F2-US6 | 作为用户，我想预览 Markdown 文档的渲染效果 | P2 |

#### 2.3.3 业务规则

| 编号 | 规则描述 |
|------|----------|
| F2-BR1 | 文档列表支持分页、排序（按时间、名称、大小） |
| F2-BR2 | 语义检索基于 LightRAG，返回相关文档片段及其来源文档引用 |
| F2-BR3 | 支持按文档类型、标签、时间范围进行筛选 |
| F2-BR4 | 文档操作（查看、删除、重新处理）需验证用户权限 |

#### 2.3.4 RAG 数据分层

文档库不只保存 Markdown 文件，还需要保存可追溯的检索数据，支撑后续文档编写和拓扑展示。

| 层次 | 内容 | 用途 |
|------|------|------|
| 原文层 | 原始文件、转换后的 Markdown、图片、表格、附件 | 保留完整资料和可回溯来源 |
| 切片层 | Chunk、章节、页码、标题路径、块级锚点、拆分关系 | 支撑精准引用、重新索引、章节级检索 |
| 语义层 | Embedding、LightRAG 索引、实体关系、Reranker 结果 | 支撑语义检索、知识图谱和相关性排序 |

每次 RAG 检索返回结果时，应同时返回内容片段、相似度/相关性评分、来源文档、来源位置和可用于拓扑图的引用关系 ID。

#### 2.3.5 接口设计

```
GET    /api/v1/library/documents              文档列表（分页、筛选）
GET    /api/v1/library/documents/:id           文档详情
GET    /api/v1/library/documents/:id/content   文档内容
GET    /api/v1/library/documents/:id/preview   Markdown 渲染预览
POST   /api/v1/library/search/keyword          关键词搜索
POST   /api/v1/library/search/semantic         语义检索
POST   /api/v1/library/documents/:id/tags      添加标签
DELETE /api/v1/library/documents/:id/tags/:tag 移除标签
```

### 2.4 F3 - 思路整理

#### 2.4.1 功能描述

用户通过多轮对话与 LLM 进行交互，采用苏格拉底式提问引导用户理清思路，最终生成结构化的文档大纲。

#### 2.4.2 用户故事

| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| F3-US1 | 作为用户，我想输入我的想法或文档背景信息，LLM 通过提问帮助我理清思路 | P0 |
| F3-US2 | 作为用户，我想在多轮对话中逐步完善我的想法 | P0 |
| F3-US3 | 作为用户，我想在对话完成后，LLM 自动生成结构化的文档大纲 | P0 |
| F3-US4 | 作为用户，我想对生成的大纲进行编辑调整 | P0 |
| F3-US5 | 作为用户，我想搜索并选择文档库中的参考资料引入讨论上下文，AI 应针对引入的资料内容进行提问 | P1 |
| F3-US6 | 作为用户，我想保存对话记录，方便后续回顾 | P2 |
| F3-US7 | 作为用户，我想查看历史头脑风暴会话列表，并可以继续之前的会话 | P1 |
| F3-US8 | 作为用户，我想在对话面板中看到已引入的参考文档列表及其数量 | P1 |

#### 2.4.3 功能流程

```
用户输入想法/背景信息
    │
    ▼
LLM 分析输入内容
    │
    ▼
LLM 生成苏格拉底式提问（理解、澄清、深化）
    │
    ▼
用户回答 ──否──▶ 用户确认思路已完整？──▶ 继续对话
    │                        │
    │                       是
    │                        │
    │                        ▼
    │               LLM 生成文档大纲
    │                        │
    │                        ▼
    │               用户审阅大纲
    │                        │
    │               ┌───────┴───────┐
    │             满意            不满意
    │               │               │
    ▼               ▼               ▼
对话完成      确认大纲         调整并重新生成
                  │
                  ▼
            大纲进入文档编写阶段
```

#### 2.4.4 苏格拉底式提问策略

| 提问阶段 | 目的 | 示例问题 |
|----------|------|----------|
| 理解阶段 | 明确用户意图 | "你希望这篇文档解决什么问题？" |
| 澄清阶段 | 消除模糊点 | "你提到的'高性能'具体指什么指标？" |
| 深化阶段 | 挖掘细节和逻辑 | "这两个概念之间的关系是并列还是因果？" |
| 结构化阶段 | 确认组织方式 | "你希望按时间线还是按主题来组织内容？" |

#### 2.4.5 实现参考

思路整理模块可参考 superpowers 开源项目中头脑风暴类 skill 的交互方式，将“开放式想法输入、多轮追问、用户确认、结构化大纲输出”设计为可复用的 Skill 流程。

该模块建议通过 Skill/MCP 编排实现以下能力：

| 能力 | 说明 |
|------|------|
| 问题生成 Skill | 根据用户输入生成澄清、追问、结构化问题 |
| 大纲生成 Skill | 将多轮对话归纳为文档标题、章节、要点和预估字数 |
| 资料引入 Skill | 将用户选择的文档库资料作为讨论上下文 |
| 大纲评审 Skill | 检查大纲是否覆盖目标读者、背景、核心观点、论证结构和交付格式 |

#### 2.4.6 大纲输出格式

```json
{
  "title": "文档标题",
  "description": "文档摘要描述",
  "sections": [
    {
      "id": "sec-1",
      "title": "第一章 概述",
      "description": "章节内容描述",
      "key_points": ["要点1", "要点2"],
      "estimated_words": 1500,
      "children": [
        {
          "id": "sec-1-1",
          "title": "1.1 背景",
          "description": "子章节描述",
          "key_points": ["要点1"],
          "estimated_words": 500
        }
      ]
    }
  ],
  "total_estimated_words": 8000,
  "reference_documents": ["doc-id-1", "doc-id-2"]
}
```

```

GET    /api/v1/brainstorming/sessions                查询会话列表
GET    /api/v1/brainstorming/sessions/:id            查询会话详情

### 2.5 F4 - 文档编写

#### 2.5.1 功能描述

基于确认的大纲，系统分章节逐步编写文档。编写前自动关联文档库中的参考资料，支持双模型对比生成，支持图片和架构图插入。

#### 2.5.2 用户故事

| 编号 | 用户故事 | 优先级 |
|------|----------|--------|
| F4-US1 | 作为用户，我想基于大纲逐章节编写文档 | P0 |
| F4-US2 | 作为用户，我想为每个章节设置限定条件（参考章节、字数限制、附加要求） | P0 |
| F4-US3 | 作为用户，我想同时使用两个 LLM 服务生成章节内容，对比择优 | P0 |
| F4-US4 | 作为用户，我想跳过某些章节，先编写指定章节 | P1 |
| F4-US5 | 作为用户，我想手动编辑 LLM 生成的章节内容 | P0 |
| F4-US6 | 作为用户，文档中需要图片时，优先使用参考文档中的图片 | P1 |
| F4-US7 | 作为用户，参考文档无图片时，可使用文生图 LLM 服务补充图片 | P1 |
| F4-US8 | 作为用户，我想通过 LLM 生成架构图并嵌入文档 | P2 |
| F4-US9 | 作为用户，我想预览完整文档的最终效果 | P1 |
| F4-US10 | 作为用户，我想将完成的文档导出为 PDF/Word/Markdown 格式 | P1 |

#### 2.5.3 功能流程

```
大纲确认

#### 2.5.4 章节编写上下文管理策略

| 上下文组成部分 | 说明 | 管理方式 |
|----------------|------|----------|
| 大纲结构 | 全局结构信息 | 始终包含，压缩表示 |
| 已完成章节摘要 | 前序章节的内容摘要 | 由 LLM 在章节完成后自动生成摘要 |
| 参考文档片段 | 与当前章节相关的文档内容 | 通过 RAG 语义检索动态获取 |
| 本章限定条件 | 用户设定的约束 | 用户输入，原样包含 |
| 跨章节引用 | 用户指定的参考章节 | 仅包含被引用章节的摘要 |

#### 2.5.5 章节状态机

为避免长文档生成过程失控，每个章节必须作为独立状态单元进行管理。

| 状态 | 含义 | 关键数据 |
|------|------|----------|
| pending | 章节未开始 | 大纲节点、章节目标、预估字数 |
| retrieving | 正在检索参考资料 | RAG 查询、候选引用片段 |
| generating | 单模型生成中 | 模型 ID、Prompt、Token 估算 |
| comparing | 双模型生成中 | 模型 A/B、两份候选内容 |
| failed | 生成失败 | 错误信息、失败原因、重试次数 |
| reviewing | 待用户审阅 | 生成内容、引用来源、差异对比 |
| accepted | 用户已确认 | 最终章节正文、采用模型、引用关系 |
| summarized | 已生成章节摘要 | 章节摘要、关键结论、可被后续章节引用的压缩上下文 |
| locked | 稳定版本 | 后续章节引用时优先使用该版本摘要 |

当 LLM 调用超时或返回错误时，章节进入 failed 状态，用户可选择重试（使用相同或不同模型）、手动编辑或跳过。

章节生成时应避免把所有已完成章节全文塞入上下文，默认只使用全局大纲、当前章节目标、用户约束、RAG 片段、已确认章节摘要和用户指定引用章节的精简内容。

#### 2.5.5.1 章节版本历史

每个章节在 accepted 状态后应保留版本历史，记录每次生成/编辑/确认的内容快照。用户可查看历史版本并回滚到任意版本。

#### 2.5.6 双模型对比机制

```
                    用户发起章节编写
                          │
                          ▼
                   ┌──────────────┐
                   │   上下文组装   │
                   └──────┬───────┘
                          │
                  ┌───────┴───────┐
                  ▼               ▼
           ┌─────────┐     ┌─────────┐
           │ LLM-A   │     │ LLM-B   │
           │ 并行生成  │     │ 并行生成  │
           └────┬────┘     └────┬────┘
                │               │
                └───────┬───────┘
                        ▼
               ┌────────────────┐
               │  对比展示界面    │
               │  A 内容 | B 内容 │
               └────────┬───────┘
                        ▼
                用户选择/合并/编辑
                        │
                        ▼
                  章节内容确认
```

双模型对比不应作为所有章节的默认行为。建议默认使用单模型生成，在关键章节、用户主动开启或质量不满意时启用双模型对比，以控制成本和降低用户审阅负担。

双模型结果应支持：

| 操作 | 说明 |
|------|------|
| 选择 A/B | 直接采用其中一个模型输出 |
| 合并 | 由 LLM 或用户将两份候选内容合并为最终版本 |
| 编辑 | 用户手动修改候选内容后确认 |
| 重新生成 | 调整约束、参考资料或模型后重新生成 |

#### 2.5.7 图片处理规则

| 场景 | 处理方式 | 优先级 |
|------|----------|--------|
| 参考文档包含相关图片 | 提取参考文档图片，嵌入文档 | 最高 |
| 需要图片但参考文档无图 | 调用文生图 LLM 服务生成 | 次高 |
| 需要架构图/流程图 | LLM 通过 MCP + Skill 调用 DrawIO 生成 PNG | 特殊 |

对于技术文档，架构图和流程图优先采用结构化图表生成方式。MVP 阶段建议优先支持 Mermaid，后续再扩展 DrawIO XML 生成、编辑和 PNG 导出能力。

#### 2.5.8 技术依赖

| 依赖 | 用途 |
|------|------|
| Mermaid | MVP 阶段架构图/流程图生成 |
| [DrawIO](https://github.com/jgraph/drawio) | 架构图/流程图生成 |
| 文生图 LLM 服务（DALL-E / Stable Diffusion 等） | 图片生成 |

#### 2.7.4 模型能力管理

模型管理不应只维护 API 地址和 API Key，还应按能力管理模型用途，便于不同功能模块选择合适模型。

| 能力标签 | 说明 | 使用场景 |
|----------|------|----------|
| chat | 通用对话 | 思路整理、多轮澄清 |

| default_for | 默认用途，如默认写作模型、默认嵌入模型 |

#### 2.7.5 Token 消耗统计

| 统计维度 | 说明 |
|----------|------|
| 按模型 | 每个 LLM 服务的 Token 用量 |
| 按功能 | 文档初始化/思路整理/文档编写各环节用量 |
| 按时间 | 日/周/月消耗趋势 |
| 按文档 | 每篇文档编写消耗的 Token 总量 |

MVP 阶段可先记录模型调用日志，包括调用时间、功能模块、模型 ID、输入/输出 token、费用估算、关联文档/章节 ID。Token 报表可在调用日志稳定后再聚合生成。

## 四、技术架构

### 4.1 推荐技术栈

| 层次 | 技术选型 | 理由 |
|------|----------|------|
| **前端框架** | Next.js 15 (App Router) | SSR/SSG 支持、React 生态、TypeScript 原生支持 |
| **UI 组件库** | Tailwind CSS + shadcn/ui | 高度可定制、现代设计、良好 TypeScript 支持 |
| **业务 API** | Next.js API Routes / Server Actions | 前后端统一框架，承载轻量业务操作 |
| **文档处理 Worker** | Python 服务 / Celery Worker | MarkItDown、OCR、文件解析等长任务更适合 Python 生态 |
| **LLM 编排 Worker** | Python 或 Node.js Worker | 承载章节生成、摘要、拆分、重试、双模型并行等异步任务 |
| **任务队列** | Redis + BullMQ / Celery / Temporal（可选） | 管理上传、转换、索引、生成等耗时任务 |
| **数据库** | PostgreSQL | 关系型数据、JSONB 支持、全文检索、成熟稳定 |
| **ORM** | Prisma | 类型安全、迁移管理、与 Next.js 生态契合 |
| **认证** | NextAuth.js + Appwrite（可选） | 本地认证 + 云端认证双模式 |
| **缓存** | Redis（可选） | 会话管理、Token 限流、热数据缓存 |
| **对象存储** | AWS S3 / MinIO（兼容 S3 协议） | 云端使用 AWS S3，离线使用 MinIO |
| **向量化/RAG** | LightRAG | 语义检索、文档向量化 |
| **文档转换** | MarkItDown | 多格式文档转 Markdown |
| **拓扑可视化** | D3.js / React Flow | 动态拓扑图渲染 |
| **架构图生成** | Mermaid + DrawIO（MCP 集成） | MVP 优先 Mermaid，后续支持 DrawIO 精细编辑和 PNG 导出 |
| **本地服务运行** | Node.js Runtime + Python Runtime | 支持非 Docker 的本机 Web 服务运行 |
| **容器化** | Docker + Docker Compose（可选） | 面向技术用户和服务器部署，提供一致运行环境 |
| **包管理** | pnpm | 快速、节省磁盘空间 |

### 4.1.1 服务拆分建议

为保证离线本地服务部署、长任务稳定性和后续扩展能力，建议采用“Next.js 主应用 + 后台 worker + RAG 服务”的轻量服务化架构。

| 服务 | 职责 |
|------|------|
| Web App | 页面、用户交互、业务 API、权限校验 |
| Document Worker | 文件上传后处理、MarkItDown 转换、OCR、Markdown 清洗、文档拆分 |
| RAG Service | LightRAG 索引、检索、删除、重建、rerank、返回引用来源 |
| LLM Orchestrator | 思路整理、章节生成、双模型对比、摘要压缩、图表生成 |
| Storage Adapter | 屏蔽本地文件系统、S3、MinIO 等存储差异 |
| Model Adapter | 屏蔽 Ollama、OpenAI-compatible、云厂商、自定义模型差异 |

Next.js API 不应直接执行大文件转换、RAG 索引和长时间 LLM 生成任务，而应创建任务并返回任务 ID，由后台 worker 异步执行，前端通过状态接口或实时通道查看进度。

## 五、部署方案

### 5.1 离线本地服务部署（优先）

离线本地服务版是本项目的优先离线交付形态。用户下载应用包后，在本机启动 Synthetix 服务，通过浏览器访问本地地址使用，例如 `http://localhost:3000`。该形态不依赖 Docker，也不采用桌面安装壳。

```
Local Service Package:
├── web-app              # Next.js Web 应用，本机启动
├── local-api            # 本地业务 API
├── document-worker      # 文档转换、OCR、拆分
├── rag-service          # LightRAG 服务或轻量本地 RAG 服务
├── llm-worker           # LLM 编排、章节生成、摘要
├── local-storage        # 用户选择的本地文档目录
├── sqlite/postgresql    # 本地数据库，MVP 优先 SQLite，增强版可用 PostgreSQL
└── ollama-adapter       # 检测和连接本机 Ollama，可选
```

本地服务版要求：

| 要求 | 说明 |
|------|------|
| 浏览器访问 | 应用启动后自动或提示用户访问 `localhost` 地址 |
| 不依赖 Docker | 普通用户无需安装 Docker 即可运行 |
| 不做桌面壳 | 不使用 Tauri/Electron 打包桌面应用，避免额外桌面端复杂度 |
| 本地数据可控 | 文档、索引、模型配置和用户数据默认保存在用户选择的本地目录 |
| 可选本地模型 | 检测 Ollama / LM Studio 等本地模型服务，未安装时允许用户配置云端 API |
| 可选高级组件 | OCR、完整 LightRAG、DrawIO 等能力可作为可选组件启用 |

### 5.2 离线 Docker 部署（可选）

Docker Compose 适合技术用户、小团队或企业内网服务器部署，用于获得更一致的运行环境和服务隔离。

```
Docker Compose:
├── nextjs-app          # 应用服务
├── postgresql          # 数据库
├── minio               # S3 兼容对象存储
├── document-worker     # 文档转换、OCR、拆分
├── rag-service         # LightRAG 服务
├── llm-worker          # LLM 编排、章节生成、摘要
├── ollama              # 本地 LLM 服务
└── redis (可选)         # 缓存服务
```

### 5.3 云端部署

```
├── Vercel / AWS EC2     # 应用托管
├── RDS PostgreSQL       # 托管数据库
├── AWS S3               # 对象存储
├── Worker Runtime       # 文档处理、RAG、LLM 编排任务
├── Appwrite             # 用户认证服务
└── 公有云 LLM API       # LLM 服务
```

## 六、项目里程碑（建议）

### 6.1 MVP 优先级

第一阶段目标是验证“可信长文档生成”核心闭环，不建议同时启动所有高级能力。

| 阶段 | 内容 | 目标 |
|------|------|------|
| V0.1 | 文档上传、MarkItDown 转换、Markdown 入库、基础文档库 | 能把资料稳定变成可管理的 Markdown |
| V0.2 | LightRAG 索引、语义检索、引用来源返回 | 能基于资料检索相关片段并追溯来源 |
| V0.3 | 思路整理、多轮问答、大纲生成和编辑 | 能从用户想法生成可用大纲 |
| V0.4 | 单模型章节生成、章节状态机、章节摘要、最终拼接 | 能按章节稳定生成长文档 |
| V0.5 | 引用关系记录、基础文档拓扑、调用日志 | 能解释生成内容用了哪些参考资料 |
| V0.6 | 双模型对比、Token 统计、导出 | 增强质量选择、成本可视化和交付能力 |
| V0.7 | 图片复用、Mermaid/DrawIO、文生图 | 增强图文和架构图能力 |
| V1.0 | 云端部署、团队协作、Appwrite/SSO、权限增强 | 支持在线部署和团队使用 |

### 6.2 开发排期建议

| 阶段 | 内容 | 周期 |
|------|------|------|
| P0 - 基础框架 | 项目初始化、本地认证、模型配置、任务队列基础 | 第 1-2 周 |
| P1 - 文档处理 | 文档初始化、MarkItDown Worker、文档库管理 | 第 3-4 周 |
| P2 - RAG 检索 | LightRAG 服务接入、索引、检索、引用来源 | 第 5-6 周 |
| P3 - 智能辅助 | 思路整理、大纲生成、Skill 流程 | 第 7-8 周 |
| P4 - 核心编写 | 单模型章节生成、章节状态机、摘要压缩、拼接输出 | 第 9-11 周 |
| P5 - 可信追踪 | 引用关系、基础拓扑、调用日志、Token 统计雏形 | 第 12-13 周 |
| P6 - 高级功能 | 双模型对比、图片/架构图、导出 | 第 14-15 周 |
| P7 - 测试部署 | 集成测试、性能优化、本地服务/Docker/云端部署验证 | 第 16-17 周 |

## 七、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| LLM API 不稳定 | 文档生成中断 | 本地模型兜底 + 重试机制 |
| 大文档上下文溢出 | 生成质量下降 | 章节化编写 + 摘要压缩策略 |
| 向量化精度不足 | RAG 检索质量低 | 支持多种嵌入模型，允许用户切换 |
| 文档拆分不合理 | 知识碎片化 | LLM 语义拆分 + 人工调整 |
| 离线模型性能有限 | 响应速度慢 | 模型量化 + 流式输出 |
