# 高质量与高性能文档语义拆分优化方案 (High-Quality & High-Performance Semantic Splitting)

本方案旨在针对知识库超大文档（80M+）在导入拆分阶段的性能问题，提供**保证拆分质量（Semantic Quality）**前提下的加速建议。

---

## 1. 核心矛盾：为什么现有的“高质量”拆分手段在超大文件上失效？

在现有代码 [semantic-splitter.ts](file:///E:/project01/src/lib/documents/semantic-splitter.ts) 中，使用的是基于 **Chat LLM 交互决策** 的语义合并方案：
1. **输入划分**：将文档初步按结构切分成数千个基础 Chunk。
2. **LLM 章节判重**：将相邻的 20 个章节标题发送给 LLM，让 LLM 逐一判断它们是否属于同一主题，并给出是否合并（`merge` 或 `keep`）的 JSON 决策。
3. **并发调用**：这会导致产生 **数百个 LLM 顺序/并发 API 请求**。

### 为什么变慢？
- **限流重试**：海量 API 请求导致 100% 触发大模型服务商的限流（429）。
- **超时降级**：最终往往因为超过 5 分钟的队列限制而报错，强制降级到基础的结构拆分（[pipeline.ts:L187](file:///E:/project01/src/lib/documents/pipeline.ts#L187)），**相当于白白等待了 5 分钟，最后拿到的依然是未优化的基础分块，质量反而受损**。

---

## 2. 优化方案：兼顾“高质量”与“高效率”的拆分技术

为了在**不降低拆分质量**（即保证语义关联性高的内容分在同一个 Chunk 中，避免上下文割裂）的前提下实现秒级处理，我们建议采用以下三种演进方案：

### 方案一：基于向量相似度（Embedding Distance）的语义分块法 (推荐：工业界标准)
这是 LangChain 和 LlamaIndex 中 `SemanticChunker` 的标准实现方式。它不依赖慢速的 Chat LLM 进行文本判断，而是直接通过**计算向量距离**来寻找主题的分界线。

#### 1. 工作原理
1. **微观切分**：首先将 Markdown 文档切分成较小的语义单元（如句子或自然段）。
2. **批量向量化**：调用 Embedding 接口获取所有句子的向量。
   > *优势*：Embedding API 支持超大 Batch（单次请求可处理 512~2048 个句子）。10,000 个句子仅需 5-10 次 API 并发请求，在 2 秒内即可完成。
3. **相似度滑动窗口**：计算相邻句子间的余弦相似度（Cosine Similarity）：
   $$Similarity(S_i, S_{i+1})$$
4. **寻找断崖点（Split Boundaries）**：
   - 设定一个阈值（如相似度低于历史滑动平均值的特定标准差，或设定 85% 阈值）。
   - 当相邻句子的相似度发生骤降时，意味着**文档探讨的主题在此处发生了切换**，在此处进行切分。
   - 相似度高的句子则自动合并到同一个 Chunk 中，直至达到 Token 上限。

```
句子索引:   S1 --- S2 --- S3 | --- S4 --- S5 --- S6 | --- S7 --- S8
余弦相似度:    0.92   0.94  [0.61]   0.89   0.91  [0.55]   0.93
主题判定:     [-- 主题 A --]  |     [-- 主题 B --]  |    [-- 主题 C --]
                           ^                     ^
                       (在此切分)             (在此切分)
```

#### 2. 为什么能兼顾质量与性能？
- **高质量**：基于向量的余弦相似度能极精准地识别出两句话是否在探讨同一概念，其语义边界识别能力不亚于 Chat LLM。
- **高性能**：生成 10,000 个句子的相似度计算是纯本地 CPU 的矩阵运算，耗时在毫秒级；仅有的 Embedding 接口调用耗时小于 3 秒。

---

### 方案二：全局结构图谱（Table of Contents）一次性决策法 (LLM 级质量)
如果仍希望保留 Chat LLM 对大纲逻辑的深层理解，我们需要改变“局部滑动窗口”的调用方式，改用**全局目录结构一次性投喂**。

#### 1. 工作原理
1. **生成全局目录树 (TOC)**：先用轻量正则提取出整个大文档的完整标题层级树（TOC）。
2. **单次/少量投喂**：将这棵目录树（一般也就几千字）一次性发送给 LLM。
   - *Prompt 示例*：“这是整个大文档的目录结构。请根据各章节之间的语义紧密程度，将它们划分成合理的主题块，并返回块的索引范围（如 `[[0, 15], [16, 42], ...]`）。”
3. **切分映射**：后端根据 LLM 返回的范围，直接在本地对基础 Chunk 进行合并。

#### 2. 为什么能兼顾质量与性能？
- **高质量**：LLM 拥有了**全局视野**，能够看到全文的脉络，避免了方案一中局部微观句意相近但宏观讨论不同主题的“假合并”现象。
- **高性能**：从 **数百次 LLM API 调用骤减为 1-2 次**，完全消除了 Rate Limit 锁死和队列超时风险，时间控制在 5 秒以内。

---

### 方案三：启发式混合拆分策略 (Heuristic Hybrid Split)
通过内置文档结构规则进行快速合并，只对模糊地带引入向量判断。

#### 1. 工作原理
1. **Markdown 语法树合并**：根据 Markdown 的标题级别（`#` -> `##` -> `###`）做树状合并。
   - 属于同一个 `##` 下的 `###` 子章节，如果它们的总字数加起来小于 `chunkMaxTokens`，**直接在本地基于语法树合并**，无需经过任何 LLM 或向量计算。
2. **边界文本校验**：只在前一个大章节的末尾段落与后一个大章节的开始段落之间，计算 Embedding 相似度，确定它们是否属于连续上下文。

---

## 3. 方案对比与实施建议

| 方案 | 拆分质量等级 | 生成速度 (80M 文档) | API 依赖成本 | 实现复杂度 |
| :--- | :--- | :--- | :--- | :--- |
| **方案一（向量相似度）** | **高 (Semantic-level)** | **极快 (~3s)** | 极低 (仅消耗极少 Embedding 额度) | 中等 (需要本地矩阵相似度计算) |
| **方案二（TOC 全局决策）** | **极高 (Global Logical)** | **快 (~5s)** | 中等 (1-2次 Chat LLM 调用) | 较低 (只需重构 Prompt 与解析器) |
| **方案三（启发式混合）** | 中上 (Rule-based) | 极快 (<1s) | 零成本 | 高 (需要写复杂的 Markdown 语法分析) |

### 🛠️ 改造路线建议
我们强烈建议采用 **方案二（TOC 全局决策） + 方案一（向量相似度）作为兜底** 的组合设计：
1. 后端提取出文档所有的 Heading 标题，拼装成完整的 Table of Contents。
2. 将 TOC 目录发送给 Chat LLM，进行一次性大块划分决策（Stage 1）。
3. 如果文档本身严重缺乏标题结构（TOC 提取失败），则自动降级到方案一，通过句向量的余弦相似度来进行本地断崖点判定（Stage 2）。

这样既保证了有结构文档的**全局逻辑质量**，又确保了无结构文档的**微观语义质量**，同时将处理耗时牢牢锁定在 **5 秒** 左右，彻底解决队列阻塞问题。
