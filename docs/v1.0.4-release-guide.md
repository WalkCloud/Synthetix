# v1.0.4 发布操作清单

**目标**：从同一个 release commit/tag 构建并验证 Windows x64 与 macOS arm64 产物；两个平台验证通过后，才把 GitHub Draft Release 发布为公开 Release。

**唯一版本源**：`package.json` 的 `version` 必须是 `1.0.4`。`src/generated/app-version.ts` 由 `pnpm run generate:meta` 生成并提交；`public/build-info.json` 是每次构建都会变化的 gitignored 动态文件，不提交、不强行跟踪。

> 本流程不包含 `feat/macos-installer` 的额外合并步骤。发布代码应先在当前已批准的源码分支上准备好，再创建一个 release commit，并让 `v1.0.4` tag 精确指向该 commit。

---

## 1. 创建唯一的 release commit

在准备发布的源码分支上完成版本和文档检查：

```bash
SOURCE_BRANCH="$(git branch --show-current)"
test -n "$SOURCE_BRANCH"

# package.json.version 应已改为 1.0.4
pnpm run generate:meta
pnpm run verify:versions

git add package.json src/generated/app-version.ts docs/v1.0.4-release-guide.md
git commit -m "release: v1.0.4"
```

不要执行以下操作：

- 不要把 `public/build-info.json` 加入 Git；它是 gitignored 的动态构建信息。
- 不要为 Windows 和 macOS 分别创建 release commit。
- 不要从不同分支或不同 commit 构建两个平台的安装包。

记录并核对唯一 release commit：

```bash
RELEASE_COMMIT="$(git rev-parse HEAD)"
git show --stat --oneline "$RELEASE_COMMIT"
git status --short
```

工作区中如有明确保留的本地改动或 stash，应保持不变；发布 commit 只应包含已批准的发布内容。

---

## 2. 先推源码分支，再推 tag

先把 release commit 推到远端源码分支：

```bash
git push origin "$SOURCE_BRANCH"
```

确认远端分支已包含同一个 commit 后，再创建 annotated tag：

```bash
git tag -a v1.0.4 "$RELEASE_COMMIT" -m "v1.0.4 — RAG cross-document integrity and macOS arm64"

test "$(git rev-list -n 1 v1.0.4)" = "$RELEASE_COMMIT"
git show --no-patch --decorate v1.0.4
```

最后推送 tag：

```bash
git push origin v1.0.4
```

> **外部副作用警告**：推送 `v1.0.4` 会立即触发 `.github/workflows/release-windows.yml`，开始 Windows 构建、签名和 GitHub Release 上传。只有在源码分支已经成功推送、tag 指向已经复核且仓库 secrets 已就绪时才能执行。准备版本和文档的任务不得代替发布操作者推送分支或 tag。

---

## 3. Windows publisher 创建或复用 Draft Release

`v1.0.4` tag 推送后，Windows workflow 在该 tag 上执行：

1. 安装依赖并生成 Prisma client。
2. 运行 `pnpm run generate:meta` 和 Next.js standalone 构建。
3. 在 clean runner 上下载固定版本的 Windows x64 Node 与 python-build-standalone，把 `workers/python/requirements.txt` 安装进 bundled Python；不得依赖预热 cache。
4. 在 Next build 后运行 `node scripts/build-installer.mjs --assemble-only --no-build`，组装并检查 `dist/app/runtime/node.exe`、`runtime/python/bin/python.exe`（兼容 flat fallback）、worker requirements、Prisma package 与 migrations。
5. 若配置了 `WINDOWS_CERT_FILE`，先把 base64 `.pfx` 解码到 runner 临时目录；若未配置则明确 warning 并继续 unsigned 构建。
6. `electron:build` 将临时证书路径和 `WINDOWS_CERT_PASSWORD` 映射为 electron-builder 标准 `CSC_LINK` / `CSC_KEY_PASSWORD` 环境变量，在构建 Windows x64 安装包时完成 Authenticode 签名；密码不会放入命令行或日志。
7. 运行 `pnpm run verify:versions`。
8. publisher 只生成并签名 `stable.json`、创建或检查 Draft Release 并上传既有产物，不接收 Windows 证书参数。
9. publisher 用 `gh release view v1.0.4 --json isDraft --jq .isDraft` 检查对应的 GitHub Release：
   - Release 不存在时，创建 **Draft Release**；
   - existing Release 只有在命令严格输出 `true` 时才复用并上传；
   - existing public Release（`false`）或任何异常/含糊输出都会拒绝上传，绝不覆盖公开 Release；
   - 将 EXE 和 `stable.json` 上传或覆盖到已确认的 Draft。

不要在 publisher 之前手动创建公开 Release，也不要使用 `gh release create` 创建非 draft Release。publisher 遇到 existing public Release 会直接失败，不能把它当作可覆盖目标。

检查 workflow 成功，并确认 Release 仍为 draft：

```bash
gh run list --repo WalkCloud/Synthetix --workflow release-windows.yml --limit 5
gh release view v1.0.4 \
  --repo WalkCloud/Synthetix \
  --json tagName,isDraft,isPrerelease,name,assets
```

必须确认：

- `tagName` 是 `v1.0.4`；
- `isDraft` 是 `true`；
- Windows 安装包文件名包含 `1.0.4`；
- 若配置了 Authenticode secrets，构建日志在 `electron:build` 开始时显示签名已启用（但不显示密码），且安装包签名验证通过；若未配置，日志包含 unsigned warning；
- `stable.json` 已存在；
- workflow 构建使用的 commit 与 `RELEASE_COMMIT` 相同。

若 workflow 因 secret 配置、网络、GitHub 服务或 runner 等**瞬态/环境问题**失败，且不需要修改源码，可以修正环境后 rerun 同一 tag 的 workflow。若修复必须修改任何源码、构建脚本或已纳入 release commit 的文件，则立即停止 v1.0.4 发布：不得移动、删除后重建或强推已共享的 `v1.0.4` tag；应提交修复、提升版本，并创建新的 release commit/tag 发布。

---

## 4. macOS 从同一个 tag 构建 1.0.4 arm64 DMG

在 Apple Silicon Mac 上获取并 checkout 已推送的同一个 tag；不要从本地分支尖端或旧工作树构建：

```bash
git fetch origin --tags
git checkout --detach v1.0.4

test "$(git rev-parse HEAD)" = "$(git rev-list -n 1 v1.0.4)"
test "$(node -p 'require("./package.json").version')" = "1.0.4"

pnpm install --frozen-lockfile
pnpm exec prisma generate
pnpm run generate:meta
pnpm run verify:versions
pnpm run build
pnpm run build-installer:mac
pnpm run electron:build:mac
pnpm run verify:versions
```

预期产物：

```text
dist/electron/Synthetix-1.0.4-arm64.dmg
```

构建后至少核对：

```bash
test -f "dist/electron/Synthetix-1.0.4-arm64.dmg"
test "$(node -p 'require("./package.json").version')" = "1.0.4"
pnpm run verify:versions
git describe --exact-match --tags HEAD   # 必须输出 v1.0.4
```

构建后的 `verify:versions` 会检查 `dist/electron/mac-arm64/Synthetix.app/Contents/Info.plist` 的 `CFBundleShortVersionString`；若 `Contents/Resources/app.asar` 可读，还会检查其中 `package.json.version`。因此它验证的是应用包内部版本，不只是 DMG 文件名。

只允许上传由 `v1.0.4` tag 实际构建且文件名、应用内部版本均为 1.0.4 的 DMG；不得用重命名旧产物的方式替代构建。

---

## 5. 上传 DMG 到同一个 Draft Release

上传前再次确认目标 Release 存在且仍是 draft：

```bash
test "$(gh release view v1.0.4 \
  --repo WalkCloud/Synthetix \
  --json isDraft \
  --jq '.isDraft')" = "true"
```

把 arm64 DMG 上传到 Windows publisher 创建或复用的同一个 Draft Release：

```bash
gh release upload v1.0.4 \
  "dist/electron/Synthetix-1.0.4-arm64.dmg" \
  --repo WalkCloud/Synthetix \
  --clobber
```

此时 Draft Release 应包含至少三个资产：

- Windows x64 EXE，文件名包含 `1.0.4`；
- `stable.json`；
- `Synthetix-1.0.4-arm64.dmg`。

核对资产：

```bash
gh release view v1.0.4 \
  --repo WalkCloud/Synthetix \
  --json isDraft,assets \
  --jq '{isDraft, assets: [.assets[].name]}'
```

如果 Draft Release 不存在，应先调查 Windows publisher 为什么没有创建它；不要改为创建公开 Release。

---

## 6. 在 Draft 阶段验证两个平台

从 Draft Release 下载候选资产，而不是使用本机构建目录中未经上传验证的副本：

```bash
rm -rf .release-verify/v1.0.4
mkdir -p .release-verify/v1.0.4
gh release download v1.0.4 \
  --repo WalkCloud/Synthetix \
  --dir .release-verify/v1.0.4
```

### Windows 验证

- [ ] 在干净的 Windows x64 环境安装 Draft 中下载的 1.0.4 EXE。
- [ ] 应用能启动并完成首次初始化。
- [ ] About/版本信息显示 `1.0.4`。
- [ ] 核对 `stable.json` 的 `version` 为 `1.0.4`，下载 URL 指向同一 `v1.0.4` Release 的 EXE，且大小、SHA-256 和签名验证通过。
- [ ] 执行发布所需的核心 RAG/document smoke test。

### macOS 验证

- [ ] 在 Apple Silicon Mac 上挂载 Draft 中下载的 `Synthetix-1.0.4-arm64.dmg`。
- [ ] 将应用安装到 `/Applications`，按当前 ad-hoc 签名说明完成首次打开。
- [ ] 应用能启动并完成首次初始化。
- [ ] About/版本信息显示 `1.0.4`。
- [ ] 执行发布所需的核心 RAG/document smoke test。

任一平台失败时，保持 Release 为 draft。不要公开不完整或未验证的 Release。

---

## 7. 双平台通过后发布 Draft

只有以下条件全部满足后才能公开：

- [ ] `v1.0.4` tag 与唯一 release commit 对应；
- [ ] Windows EXE 和 macOS DMG 均从该 tag 构建；
- [ ] Draft 中存在 EXE、`stable.json` 和 1.0.4 arm64 DMG；
- [ ] Windows 安装验证通过；
- [ ] macOS 安装验证通过；
- [ ] Release notes 已复核；
- [ ] `isDraft` 当前仍为 `true`。

发布命令：

```bash
gh release edit v1.0.4 \
  --repo WalkCloud/Synthetix \
  --draft=false
```

发布后确认：

```bash
gh release view v1.0.4 \
  --repo WalkCloud/Synthetix \
  --json tagName,isDraft,isPrerelease,publishedAt,assets
```

预期 `isDraft` 为 `false`，三个必需资产仍完整。只有此时公开的 `stable.json` 才进入正式更新通道。
