# v1.0.4 发布操作清单

**目标**：发布包含 RAG cross-document integrity 修复 + macOS 打包能力的 v1.0.4。
**涉及两台电脑**：Windows（验证 + 编程 + 触发 CI）和 macOS（已构建 DMG）。

---

## 阶段 1：Windows 验证（在 Windows 电脑上做）

> 目的：确认 `feat/macos-installer` 分支的 macOS 改动**没有破坏 Windows 打包**。
> 这是合并到 main 前的必要检查。

### 1.1 拉取代码

```powershell
git clone https://github.com/WalkCloud/Synthetix.git   # 如果已克隆，跳到 cd + git fetch
cd Synthetix
git fetch origin
git checkout feat/macos-installer
npm install
```

### 1.2 验证 Windows 打包不受影响

```powershell
# 1. 语法检查（重构后的 build-installer.mjs 能正确 import 共享库）
node --check scripts/build-installer.mjs
node --check scripts/build-electron.mjs
node --check scripts/lib/bundle-assembly.mjs

# 2. 跑 --assemble-only（验证共享组装逻辑在 Windows 上字节一致）
npx prisma generate
npm run build
node scripts/build-installer.mjs --assemble-only --no-build

# 3. 完整 Windows 构建（需要 Inno Setup + runtime）
npm run electron:build
```

**预期**：产出 `dist/electron/Synthetix Setup 1.0.3.exe`（或类似 NSIS 安装包）。

> ⚠️ 详细验证步骤见仓库里的 `docs/windows-regression-checklist.md`。

### 1.3 如果 Windows 验证通过 → 回到 macOS 或继续阶段 2

---

## 阶段 2：合并到 main + 打 tag（可以在 Windows 或 macOS 上做）

> 目的：把验证过的代码合并到 main，然后切 v1.0.4 tag。

### 2.1 合并到 main

```bash
git checkout main
git pull origin main
git merge feat/macos-installer --no-ff -m "merge: feat/macos-installer — macOS packaging + RAG integrity for v1.0.4"

# 确认合并后 main 能正常构建
npm install   # 如果有新依赖
npm run build # 确认 .next 能构建
```

> `--no-ff` 保留合并历史，方便回溯。如果合并冲突（应该不会，因为 mac 分支不落后于 main），解决后继续。

### 2.2 Bump 版本号到 1.0.4

```bash
# 编辑 package.json，把 "version": "1.0.3" 改成 "version": "1.0.4"
# 然后生成派生文件
npm run generate:meta   # 把版本号传播到 src/generated/app-version.ts + public/build-info.json

# 确认版本一致性
npm run verify:versions  # 应输出 "all version sources consistent"

git add package.json src/generated/app-version.ts public/build-info.json
git commit -m "release: v1.0.4"
```

### 2.3 打 tag 并推送

```bash
git tag -a v1.0.4 -m "v1.0.4 — macOS packaging + RAG cross-document integrity"
git push origin main
git push origin v1.0.4
```

> ⚠️ **推送 v1.0.4 tag 会自动触发 Windows CI 构建**（`release-windows.yml`）。它会：
> 1. 构建 Windows 安装包
> 2. 用 Ed25519 密钥签名更新清单
> 3. 上传到 GitHub Release
> 确保 `SYNTHETIX_SIGNING_KEY` secret 已配置在仓库的 Settings → Secrets 里。

## 阶段 3：上传 macOS DMG（在 macOS 电脑上做）

> 目的：把已构建好的 macOS DMG 上传到 v1.0.4 的 GitHub Release。
> Windows CI 只构建 Windows 安装包，macOS DMG 需要手动上传。

### 3.1 重新构建版本号为 1.0.4 的 macOS DMG

> ⚠️ 当前 DMG 是 `Synthetix-1.0.3-arm64.dmg`（版本号 1.0.3）。
> v1.0.4 的 Release 里需要文件名和内部版本都是 1.0.4。

```bash
# 在 macOS 上，切到合并后的 main + v1.0.4 tag
git checkout main
git pull origin main
git checkout v1.0.4   # 确保代码和 tag 一致

# 重新构建（版本号已是 1.0.4，因为阶段 2.2 改过）
npm run electron:build:mac
```

产出：`dist/electron/Synthetix-1.0.4-arm64.dmg`

### 3.2 上传 DMG 到 GitHub Release

```bash
# 安装 gh CLI（如果没装）：brew install gh
gh release upload v1.0.4 "dist/electron/Synthetix-1.0.4-arm64.dmg" --repo WalkCloud/Synthetix
```

> 如果 Release 还没创建（CI 还在跑），先手动创建：
> ```bash
> gh release create v1.0.4 --repo WalkCloud/Synthetix --title "Synthetix 1.0.4" --notes "Release notes here"
> gh release upload v1.0.4 "dist/electron/Synthetix-1.0.4-arm64.dmg" --repo WalkCloud/Synthetix
> ```

### 3.3 验证 Release 资产完整

GitHub Release v1.0.4 最终应该包含：
- `Synthetix Setup 1.0.4.exe`（Windows 安装包，CI 自动上传）
- `stable.json`（更新清单，CI 自动上传）
- `Synthetix-1.0.4-arm64.dmg`（macOS 安装包，手动上传）

## 常见问题

**Q: 如果 Windows CI 构建失败怎么办？**
A: 检查 GitHub Actions 日志。最常见原因：`SYNTHETIX_SIGNING_KEY` 未配置（manifest 签名失败）、或 build-installer.mjs 重构在 Windows 上有路径问题。

**Q: macOS DMG 的版本号必须是 1.0.4 吗？**
A: 是的。版本号不一致会导致应用内更新检测混乱。阶段 3.1 从 v1.0.4 tag 构建会确保版本正确。

**Q: 我能跳过 Windows 验证直接发布吗？**
A: 不建议。build-installer.mjs 的重构虽然设计为纯提取（已逐行验证），但 Windows 文件系统行为（路径分隔符、文件锁）可能暴露问题。阶段 1 的验证就是为了catch这个。

**Q: feat/macos-installer 分支要不要删？**
A: 合并后可以删，但建议保留到 v1.0.4 验证通过后再删：
> ```bash
> git push origin --delete feat/macos-installer   # v1.0.4 验证通过后
> ```
