# Synthetix 基线产品规格 Spec

## Why
当前仓库已有完整的需求分析文档和静态原型，但缺少一份可直接驱动后续实现拆解的正式规格基线。需要把需求分析中的目标、范围、模块边界和 MVP 优先级收敛为统一的变更单，作为后续实现与验收的唯一入口。

## What Changes
- 新增 Synthetix 产品基线规格，明确 MVP 闭环为“资料上传与初始化 -> 文档库检索 -> 思路整理 -> 大纲确认 -> 分章节写作 -> 引用追踪与导出”
- 新增模块级能力定义，覆盖仪表盘、异步任务通知、文档初始化、文档库、思路整理、文档编写、文档拓扑、模型管理、用户管理
- 新增系统架构约束，明确 Next.js 仅承载界面与轻量 API，长任务统一下沉到独立 worker 或服务
- 新增部署基线，明确离线本地 Web 服务为第一优先级，Docker 和云端部署作为后续扩展
- 新增非功能要求基线，覆盖性能、可靠性、安全、扩展性和兼容性

## Impact
- Affected specs: 产品范围定义、MVP 阶段规划、异步任务架构、文档处理链路、RAG 检索、写作状态机、模型能力治理、认证与系统设置
- Affected code: `prototype/` 静态原型、未来 `app/` 页面与 API、后台 worker、RAG 服务、模型适配层、存储适配层、任务队列、认证模块

## ADDED Requirements
### Requirement: MVP 核心闭环
系统 SHALL 支持从参考资料导入到长文档导出的最小可用闭环，并保证每一步都可被用户感知、追踪和继续操作。

#### Scenario: 用户完成一次端到端写作流程
- **WHEN** 用户上传参考文档并完成初始化
- **THEN** 系统生成可检索的 Markdown 文档与索引数据
- **AND** 用户可以在思路整理中引用这些资料生成结构化大纲
- **AND** 用户可以基于大纲逐章节生成、审阅、确认正文
- **AND** 系统保留章节引用来源并支持最终导出

### Requirement: 异步任务执行边界
系统 SHALL 将文档转换、索引构建、章节生成、摘要压缩和双模型并行生成视为异步任务，Web 应用只负责任务创建、状态查询和结果展示。

#### Scenario: 用户触发长耗时任务
- **WHEN** 用户上传大文件或发起章节生成
- **THEN** API 返回任务标识而不是同步等待任务完成
- **AND** 后台 worker 异步处理任务
- **AND** 前端通过轮询或实时通道展示进度与结果通知

### Requirement: 文档初始化与可追溯索引
系统 SHALL 支持将多格式输入文档转换为 Markdown，并保留支撑后续引用追踪的 provenance 元数据。

#### Scenario: 用户上传异构文档
- **WHEN** 用户上传 PDF、DOCX、PPTX、XLSX、HTML、EPUB 或图片文档
- **THEN** 系统校验格式与大小限制
- **AND** 使用转换链路生成标准 Markdown
- **AND** 在拆分和向量化后保留来源文件、页码、标题路径、块 ID、拆分关系等元数据

### Requirement: 文档库检索能力
系统 SHALL 提供关键词检索与语义检索两种文档发现方式，并返回可直接用于写作与拓扑展示的引用信息。

#### Scenario: 用户检索资料
- **WHEN** 用户在文档库中执行关键词或语义查询
- **THEN** 系统返回文档或片段结果
- **AND** 每条结果包含来源文档、来源位置、相关性信息与引用关系标识

### Requirement: 思路整理与大纲确认
系统 SHALL 通过多轮苏格拉底式问答帮助用户澄清写作目标，并输出可编辑、可确认的结构化大纲。

#### Scenario: 用户从模糊想法生成大纲
- **WHEN** 用户输入背景信息并与系统进行多轮对话
- **THEN** 系统按理解、澄清、深化、结构化阶段逐步提问
- **AND** 在用户确认思路完整后生成结构化大纲
- **AND** 用户可以编辑并确认大纲进入写作阶段

### Requirement: 分章节写作状态机
系统 SHALL 以章节为最小状态单元管理长文档生成过程，避免上下文溢出并支持失败恢复。

#### Scenario: 用户生成单个章节
- **WHEN** 用户为某章节设置约束并启动生成
- **THEN** 系统按 `pending -> retrieving -> generating|comparing -> reviewing -> accepted -> summarized -> locked` 管理状态流转
- **AND** 失败时进入 `failed` 状态并提供重试、改模型、手动编辑或跳过能力
- **AND** 后续章节默认消费已确认章节摘要而不是全文

### Requirement: 模型与部署模式治理
系统 SHALL 将模型配置、能力标签和部署模式作为平台级治理能力，确保离线优先且可扩展到云端。

#### Scenario: 用户配置模型与运行环境
- **WHEN** 用户在本地离线模式下首次使用系统
- **THEN** 系统允许配置本地认证、本地或云端模型、本地存储和本地数据库
- **AND** 模型配置记录 `chat`、`writing`、`embedding`、`rerank` 等能力标签
- **AND** 核心流程不依赖 Appwrite 或公网服务

## MODIFIED Requirements
### Requirement: 部署优先级
系统 SHALL 以本地 Web 服务部署为默认交付方式，并将 Docker Compose 与云端托管视为增强部署选项，而不是 MVP 的前置依赖。

#### Scenario: MVP 部署决策
- **WHEN** 团队规划首个可交付版本
- **THEN** 默认目标是普通用户可在本机启动本地服务并通过浏览器访问
- **AND** 不要求桌面壳包装
- **AND** Docker 与云端方案不阻塞本地 MVP 闭环落地

## REMOVED Requirements
### Requirement: 桌面壳安装版
**Reason**: 需求分析已明确不采用 Tauri 或 Electron 路线，避免在早期引入额外桌面端复杂度。
**Migration**: 所有离线使用场景统一收敛到本地 Web 服务运行方案，后续仅保留本机服务启动与浏览器访问路径。
